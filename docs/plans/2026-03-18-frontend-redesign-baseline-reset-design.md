# Frontend Redesign Baseline Reset Design

**日期：** 2026-03-18

> 本文档覆盖同日上一版“器物感静奢·工作台版”视觉方向，用于重新定义 Codex Pool 的前端设计基线。

## 背景

上一版虽然已经拿掉了一部分发光、漂浮、玻璃和模板化背景，但核心问题没有被真正解决：

- 结构仍然是典型的 SaaS 模板结构：大标题、按钮组、指标卡、侧边白卡。
- 页面仍以“卡片岛”的方式组织信息，缺少连续工作面的整体秩序。
- 视觉判断仍然围绕“高级感材质”展开，而不是围绕“专业工具的编排质量”展开。
- 因此即使光效减少了，页面依然像“被 AI 精修过的后台模板”。

这次需要重定的，不是某种材质或颜色，而是整套前端的组织语言。

## 现状问题

### 1. 结构仍然像模板首页

- dashboard 仍是大标题加操作按钮，再接四张指标卡和若干白面板。
- auth 仍然有明显的“主登录面板 + 次说明面板”的模板痕迹。
- workspace/settings 也在沿用“多个圆角白卡并列”的安全做法。

### 2. 信息被切成太多独立岛屿

- 每一个功能块都像独立卡片，页面缺少连续的主工作面。
- 主次关系更多依赖背景和边框，而不是依赖位置、密度、分隔和阅读顺序。
- 结果是界面可用，但整体判断力不强，看上去像组件拼出来的。

### 3. “精致感”仍然靠包装而不是靠编排

- badge、标题、按钮、面板标题仍然带有一层“我在展示设计感”的语气。
- 大量统一圆角、统一 padding、统一白底，会形成安全但空泛的模板感。
- 这种精致感不够耐看，也不适合高频 B2B 控制台。

### 4. 控件和概览模块不够像真实工具

- 按钮、筛选器、表格工具栏和状态块都还偏“组件展示”而不是“工作控件”。
- KPI 卡片像首页素材，而不是总览读数。
- hover、焦点和表面反馈还不够“压感化”和“软件化”。

## 目标

1. 把前端基线从“静奢 SaaS”彻底换成“精编仪表台”。
2. 让页面先像专业工作软件，再像被设计过的产品。
3. 让高级感来自结构秩序、阅读路径、细节分寸和控件手感，而不是材质表演。
4. 建立一套能长期约束 dashboard、workspace、settings 的稳定准绳。

## 非目标

- 不做营销站式品牌改造。
- 不做强工业、强监控屏、强实验室 UI。
- 不为了去 AI 味而回到传统企业 OA 的厚重和陈旧。
- 不推翻 React + Tailwind + shadcn 的现有技术基底。

## 方案对比

### 方案 A：档案台秩序

特点：

- 以文档感和目录感建立秩序
- 更温和、更安静、更偏信息组织
- 适合 settings、detail、日志等页面

优点：

- 最稳，最容易去掉模板味
- 对中文长文案和高频任务有韧性

缺点：

- 概览页可能显得过平
- 容易缺少产品自己的辨识度

### 方案 B：精编仪表台

特点：

- 像被认真编排过的专业桌面控制台
- 以连续工作面、分隔纪律、注释式信息组织建立质感
- 允许少量作者性，但绝不舞台化

优点：

- 最适合 Codex Pool 的高频 B2B 控制台属性
- 能同时兼顾辨识度、精致感和工具感

缺点：

- 对布局判断要求更高
- 如果执行不稳，容易落回“普通管理台”

### 方案 C：静默工册

特点：

- 极度克制、接近系统手册式 UI
- 高密度、低装饰、几乎不强调情绪

优点：

- 最不可能产生 AI 模板感
- 最接近纯工具效率

缺点：

- 太容易过冷、过硬
- 容易丢掉产品质感和记忆点

## 结论

采用 **方案 B：精编仪表台 · 连续工作面版**。

这套方向的重点不是“更漂亮”，而是“更像一个长期被专业用户依赖的工具”：

- 页面是连续被组织起来的，不是由卡片拼起来的
- 信息层级像经过编辑，而不是像组件默认堆叠
- 控件像真实工具，有压感、有秩序、有明确主次

一句话概括：

**用连续工作面替代卡片岛，用编辑式层级替代模板式高级感。**

## 新的设计北极星

Codex Pool 的前端应该更接近：

- 被精修过的桌面控制台
- 有编排判断的专业工作界面
- 连续、安静、可信、耐用的系统软件

而不是：

- 高级感 SaaS 模板
- 登录页和 dashboard 都像“设计展示页”
- 大白卡、圆角拼图、KPI 首页和说明面板堆叠

关键词：

- 连续工作面
- 编辑式秩序
- 控件压感
- 克制而不乏味
- 专业而不沉闷
- 精致但不表演

## 设计基线

### 1. 结构优先

- 页面优先构建一块连续主工作面，再决定哪里需要局部分区。
- 主次关系优先通过位置、宽度、分隔和节奏建立，而不是通过“再包一层卡片”建立。
- 除非语义必须独立，否则避免同层出现多个等权大白面板。

### 2. 页面编排

- dashboard 默认使用“头部概览条 + 主分析区 + 窄侧注区”的版式，不再使用 hero + card grid。
- workspace 默认是“主任务面 + 次级上下文依附”，次级内容不再和主任务做同权竞争。
- settings/detail 默认是“节制的文档式页面”，更像系统设置而不是产品首页。

### 3. 排版

- 取消海报式大标题、密集大写 eyebrow、过强的 tracking 操作。
- 标题强调“判断力”而不是“姿态感”，行长、段距、密度应像软件界面而不是品牌站。
- 指标、状态、读数更像仪表读数或软件统计，不像营销数卡。

### 4. 表面与形状

- 大面积表面从“面板”改成“工作底板”；局部区块才允许形成轻度包裹。
- 整体圆角再收一档，统一从“软”转向“稳”。
- 阴影只能做接触层级提示，不能制造漂浮和奢华感。
- 允许细分隔线、区块底色、局部 inset，但不允许 glow、halo、雾化白膜。

### 5. 控件与微交互

- 按钮、输入框、筛选器和工具栏要像被使用的器件，而不是摆拍组件。
- hover 不上抬，active 更沉，focus 更清楚，状态切换更像压下一个真实控制件。
- 动效应是软件反馈，不是展示性 reveal；速度更短、更干净、更靠近“切换状态”。

### 6. 页面分型

#### Auth

- 单一登录工作面，品牌信息只做边注或页脚说明。
- 去掉双 panel、卖点 bullet、模板化欢迎区。
- 表单主角必须清楚，辅助说明必须收在次要层级。

#### Dashboard

- 更像操作总览台，不像 KPI 首页。
- 核心是总览读数、过滤上下文、趋势与异常，而不是展示“精致卡片”。
- 指标可以存在，但必须从“卡片拼图”改成“连续读数编排”。

#### Workspace

- 让主任务区成为唯一锚点。
- 错误、引导、状态、帮助都以依附方式出现，避免多块等权面板并列。

#### Settings / Detail

- 全站最安静、最稳定、最可预测。
- 通过结构和分组形成高级感，而不是通过品牌材质形成高级感。

## 明确删除的旧语言

以下内容不再作为默认语言继续使用：

- dashboard 大标题 + 三按钮 + 四张 KPI 卡片
- auth 双面板堆叠和 bullet 列表
- 多块同尺寸圆角白卡并列
- logo halo、卡片高光、按钮上抬、表头渐变
- 大量 badge 化标签和过度大写 eyebrow
- “安静奢华”式表面词汇和材质包装感

## 可保留资产

- `page-archetypes.ts` 的页面分型抽象
- `WorkspaceShell / ReportShell / SectionHeader / StandardDataTable` 的结构价值
- 现有管理台的信息架构、路由与大部分任务路径

需要重写的，是界面组织语言与基础控件的表达方式，而不是信息架构本身。

## 首轮落地范围

1. 设计 tokens 从“材质命名”切换到“结构与工作面命名”
2. 共享壳层从 card shell 改成 continuous shell
3. dashboard 首屏改成连续总览编排
4. auth 重做为单一登录工作面
5. 控件、表格、筛选器统一改成更稳的器件语言
6. import jobs / models / config 作为工作台样板页完成收敛

## 验证标准

- 第一眼不再像 AI 生成的高级 SaaS 模板。
- 用户看到的是结构、任务和读数，而不是卡片、材质和标题姿态。
- dashboard、workspace、settings 三类页面的“工作软件感”显著增强。
- 中文长文案、窄屏和 reduced-motion 下仍然成立，不靠装饰掩盖问题。

## 结论

这次不是继续给上一版“去点 AI 味”，而是直接更换视觉语法。

新的基线必须从“材质感驱动的精致”切换到“编排驱动的专业”。只有这样，后续每一页的改动才不会继续长成模板味很重的白卡后台。
